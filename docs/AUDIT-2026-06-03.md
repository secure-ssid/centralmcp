## 🔴 Critical — fix first (device-safety)

### C1. Destructive ops run **without confirmation** (elicitation is dead code)
`mcp_servers/ops.py` — `poe_bounce` (218), `port_bounce` (249), `reboot_device` (320), `disconnect_client` (364)

All four DESTRUCTIVE tools call `ctx.elicit(message=..., schema={raw dict})`. FastMCP's `Context.elicit` requires a **Pydantic model**, not a dict, so it raises — and the broad `try/except` swallows it, so the tool **proceeds and executes the destructive action**. Even with a correct schema, the accept path reads `getattr(result,"data",{}).get("confirm")`, but `ElicitationResult.data` is a Pydantic model (no `.get`) → also raises → also swallowed → proceeds.

**Fix:**
```python
from pydantic import BaseModel
class _Confirm(BaseModel):
    confirm: bool = False
result = await ctx.elicit(message=..., schema=_Confirm)
if result.action != "accept" or not result.data.confirm:
    return {"status": "CANCELLED"}
```
Remove the broad `try/except` around the decision so confirmation **fails closed**.

### C2. The 4 destructive ops tools are **unreachable through `tool_router`** (the recommended entrypoint)
`mcp_servers/tool_router.py:163` — `invoke_tool`

`reboot_device`/`port_bounce`/`poe_bounce`/`disconnect_client` are `async def` with a required `ctx: Context` first param. `invoke_tool` calls `tool.fn(**args)` directly (bypassing FastMCP's ctx injection) and then does `asyncio.get_event_loop().run_until_complete(result)` on the **already-running** loop → fails. Net: these tools either error or never run via the router.

**Fix:** dispatch through FastMCP (`await mod.mcp._tool_manager.call_tool(name, args, context=...)`), or detect `inspect.iscoroutine(result)` and run it on a dedicated loop in a worker thread:
```python
if inspect.iscoroutine(result):
    import concurrent.futures
    with concurrent.futures.ThreadPoolExecutor(max_workers=1) as ex:
        result = ex.submit(asyncio.run, result).result()
```

> Combined effect: via the router the destructive tools don't work; called directly they skip confirmation. Fix both before exposing this server to any LLM-driven client.
